# Relay rejected every mod-derived room code (and local rendering hid it)

| Field | Value |
|---|---|
| Date | 2026-04-19 |
| Area | relay |
| Tags | room-code, validation, regex, chat, presence, silent-drop |
| Status | confirmed |

## Symptom

Presence list never populated. Chat appeared to work — your own messages showed up in your history — but no presence broadcast ever arrived, and a second client in the same room never echoed messages either. The relay's stdout was completely silent for the affected client after the initial WebSocket handshake.

The smoking-gun line was only visible after we added the persistent sidecar log (`%LOCALAPPDATA%\OSPlus\sidecar.log`):

```
[WS] Joining room: AAYLD5BAT1 as 632680c154686dedd652
[WS] Server error: invalid room code
```

## Root cause

Two flavors of room code coexist in the system, and the relay's validator was scoped for only one of them:

1. **Human-shareable codes** generated by `generateRoomCode()` on the relay: 4 chars from the Crockford-ish alphabet `[A-Z2-9]` (no `0/1/I/O` to avoid look-alikes when typed). These are for a future "share a code with a friend" flow.
2. **Mod-derived codes** built by `mod/OSPlus/scripts/chat.lua` `seedToCode()`: an 8-char encoding of the server-replicated match seed, plus `T<team>` to scope each room to a single team. Concrete shape: `AAX45ABAT1` — 10 chars, and the trailing team digit can be `0` or `1`, neither of which is in `[A-Z2-9]`.

The relay's `ROOM_CODE_RE` was `/^[A-Z2-9]{4}$/`, which is correct for flavor (1) and fails for flavor (2) on **both** axes (length and alphabet). Every join from the mod was rejected with `{type:"error", error:"invalid room code"}`.

What made the bug invisible for so long:

- Lua renders the local player's outbound chat into the local history *before* round-tripping through the relay. So the chat widget always showed your own messages whether the room join succeeded or not.
- The relay drops rejected joins silently from the stdout perspective for that connection — there's no `[ROOM]` line because `addToRoom` never ran.
- The sidecar's `console.log` originally went nowhere (the process is launched hidden via `wscript.exe`), so the `Server error: invalid room code` line was lost. We only saw it after adding the file-based sidecar log earlier in the same debugging session — see [`sidecar-ws-keepalive.md`](sidecar-ws-keepalive.md), which added that logging for an unrelated reason but turned out to be the load-bearing fix that made *this* bug visible.

## Fix

Relax the relay's regex to accept both flavors:

```js
// server/index.js
const ROOM_CODE_RE = /^[A-Z0-9]{4,16}$/;
```

The validator's job is to reject control chars, whitespace, and smuggled delimiters (newlines that would corrupt the `\n`-joined presence wire format). It is **not** the right place to enforce a specific encoding — `generateRoomCode` already constrains its own output, and the mod owns its own derivation. Bumped `mod/OSPlus/scripts/config.lua` `VERSION` to `v18-relay-room-code-regex` and redeployed via `server/deploy/ship.ps1`.

No client-side change was needed; the existing `seedToCode` output is already alphanumeric and ≤16 chars.

## Lesson

Two lessons, one technical and one process:

- **Don't validate at the wrong layer.** `ROOM_CODE_RE` was conflating "what does my own generator emit" with "what is a structurally valid code on the wire". Those are different questions. The wire validator should be the loosest constraint that still protects the wire format (no control chars, no newlines, sane length). Generators can be stricter for their own UX reasons.
- **Local-only rendering hides round-trip failures.** Because the chat widget rendered local sends locally, every test of "did chat work?" was unwittingly a test of "does Lua's local history append work?" The fact that the relay returned an `error` frame was meaningless until we could *see* it. When you have a local-render-then-broadcast pattern, you need an explicit "this message was acknowledged by the relay" signal — or at minimum, surface relay error frames into the visible UI so the failure can't hide. (Followup work, not done yet: route relay `error` frames into the chat history with a distinct style.)
