# Learnings

Every non-trivial finding from working on OSPlus lands here. The discipline is enforced by `.cursor/rules/learnings-discipline.mdc`.

## What counts as a learning

Write an entry whenever **at least one** of these is true:

- The work involved more than ~30 minutes of debugging.
- A new engine/game fact was discovered (UFunction signature, BP property, lifecycle quirk).
- A new failure mode was found (a thing that crashed, broke silently, or behaved unexpectedly).
- A new tool, harness, or workflow step was introduced or significantly changed.
- A previously-believed fact turned out to be wrong.

If you're unsure, write the entry. The cost of a 5-minute log is far less than the cost of a future agent re-deriving the answer.

## How to write one

1. Copy `_TEMPLATE.md` to `<short-slug>.md` in this directory. Use kebab-case. The slug is what someone will grep for in six months.
2. Fill in every field. Empty sections mean the learning is incomplete.
3. Update any other doc the learning invalidates (`KNOWLEDGEBASE.md`, `AGENTS.md` toolchain section, etc.) — the learning entry is *additive*, not a substitute for fixing stale docs.
4. Link from the relevant code (a comment pointing to `docs/learnings/<slug>.md`) when the lesson is about a specific code location.

| 2026-04-28 | [profile-tick-userdata-allocation-leak](profile-tick-userdata-allocation-leak.md) | mod | Per-tick `identity.resolveSteamId()` call from `profile.tick` allocates ~3 UE4SS-tracked userdata wrappers per frame at 30 Hz forever, even after the first profile_upsert successfully landed and snapshot fields are session-immutable (Prometheus ID, display name, Steam ID, platform — none can change in a session). After ~17 minutes idle in lobby (~30,000 ticks × 3 = ~92,000 short-lived UE allocations), UE4SS's userdata-tracking table dereferences a stale slot returning `0xFFFFFFFFFFFFFFFF` (canonical iterator-at-end / invalid-handle sentinel) and access-violates inside its own machinery at `UE4SS.dll+0x9211B2`. UE4SS's vectored exception handler catches the fault before Windows Error Reporting engages, writing `Binaries\Win64\crash_<timestamp>.dmp` (standard MDMP format). Fix is `if lastSnapshot then return end` at the top of `M.tick` — not "cache the SteamID more aggressively." Three transferable rules: (1) per-tick UE-reflected calls allocate, treat them like per-tick `malloc`; (2) cache push-down is the callee's job; (3) for session-immutable values the right caching policy is "never refresh," polling itself is the bug. Codified in `.cursor/rules/mod-architecture.mdc` § "Per-tick discipline" + `docs/architecture/mod-scripts.md`.
