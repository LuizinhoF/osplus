## Context

OSPlus today binds profile rows to **claimed SteamID** (per `_archive/vision-v1-superseded.md` → Lock 1, archived 2026-04-23 along with the rest of the vision doc). The `in-game-profile-mvp` feature is the first to actually require a persistent profile row, which is what forces this decision now.

Two new constraints reshape the options space relative to the archived position:

1. **Maintainer requirement: platform-agnostic.** Omega Strikers may be playable from launchers other than Steam (Epic, Discord, future console ports, GeForce Now). SteamID is platform-specific by construction; binding the canonical profile row to it forecloses non-Steam players. The archived position implicitly assumed Steam-only and didn't name this.
2. **Tracker-ecosystem interop is a real future requirement, not hypothetical.** Every Omega Strikers tracker (`stats.omegastrikers.gg`, `clarioncorp.net`, `strikr.gg`, `omegastrikers.stlr.cx`) keys off the Odyssey backend's "Prometheus ID" — a 24-char hex MongoDB ObjectID — *not* SteamID. An OSPlus profile keyed on SteamID cannot join its own captures (e.g., per-match redirects from `in-game-profile-mvp`'s capture surface) against tracker-ecosystem aggregate stats without doing Prometheus-ID resolution anyway. See `docs/learnings/os-prometheus-api-ecosystem.md` and KB → *Player Identity Reference*.

Pass 3 of the in-game-profile-mvp feasibility (committed in this branch) characterized the runtime path to the local Prometheus ID end-to-end. The runtime data model is documented in `docs/learnings/os-runtime-data-model.md`. The path that was previously thought to be blocked ("`PMPlayerModel` UFunctions are not trivially callable from Lua") turned out to be a calling-convention misunderstanding — the call shape is now known. **The decision is no longer blocked by feasibility; it is blocked only on this ADR.**

## Options considered

Three independent sub-decisions are bundled in this ADR because they share a single forcing feature:

- **(P)** Primary binding key — what column is the row's identity?
- **(R)** Resolution path — how do we get that identity from the running client?
- **(T)** Trust posture — do we verify the claim, or trust it as-reported?

P and R are presented as independent options below. T is a separate section because both posture options apply on top of any P/R combination.

### Primary binding key — Options

#### P-A — Trust-on-claim SteamID (archived position)

- **What it is** — Profile row keyed on `steamId TEXT PRIMARY KEY` (17-digit decimal). Sidecar reads `PMIdentitySubsystem:GetSteamId()` (already implemented in `identity.lua`); claims it to the relay at handshake; relay trusts the claim.
- **Pros** — ~Zero implementation cost (already partially built). Stable per-player, cross-session.
- **Cons** — **Steam-only by construction.** Non-Steam launches either can't have a profile or get a synthetic fallback that has to be reconciled later. Cross-tracker interop blocked unless we *also* resolve the Prometheus ID — i.e., doing this work later instead of now. Schema migration from `steamId`-keyed rows to a different primary key later is high-cost (cascading FK rewrites).
- **Cost to build** — Near zero.
- **Cost to change later** — High. Migrating a user base off this key requires either resolving Prometheus IDs at migration time (the same work this ADR is asking us to do now) or accepting a discontinuity in profile history.

#### P-B — Prometheus ID

- **What it is** — Profile row keyed on `prometheusId TEXT PRIMARY KEY` (24-char hex). SteamID retained as a secondary indexed column (`steamId TEXT NULL` — null for non-Steam players).
- **Pros** — Platform-agnostic (the requirement that rules out P-A). Aligns with the tracker ecosystem's keying convention, so cross-tracker enrichment becomes a possibility rather than a port. Resolves a constraint we know will eventually bite (non-Steam OS launches) before any user-visible feature is built on the wrong key.
- **Cons** — Requires resolution-path work (see R-options below). Requires the runtime data path that was previously thought blocked (now unblocked per Pass 3 / `os-runtime-data-model.md`). Adds one column over P-A's schema.
- **Cost to build** — Small. Identity-resolution path is now characterized; ~50 lines of Lua + schema additions.
- **Cost to change later** — Low. SteamID retained as secondary, so cross-references stay reachable; switching primary key away from Prometheus ID is unlikely (the only other candidate is SteamID, which P-A rejects).

#### P-C — Composite key (Prometheus ID + SteamID + per-install UUID)

- **What it is** — No single primary key; row identity is the tuple. Each profile row carries any subset of the three available identifiers.
- **Pros** — Maximally flexible. Future identifiers (Epic ID, console IDs) bolt on without schema migration.
- **Cons** — Foreign-key shape ambiguous from day one — every dependent feature has to choose which column to FK against. Profile-row uniqueness becomes a definitional question (two rows with overlapping but non-identical tuples — same player or not?). Premature flexibility for a problem that doesn't exist yet (only Steam launches are observed in the field today).
- **Cost to build** — Medium. Schema design plus uniqueness-rule design plus FK guidance.
- **Cost to change later** — Lowest of the three (it's already flexible) — but the cost of *living with the complexity* every day is high.

### Resolution path — Options (assume binding-key Option P-B)

#### R-A — Synchronous polling

- **What it is** — `identity.lua` polls `FindFirstOf("PMPlayerModel"):GetCachedMeResponseV1(false, nil)` on each tick. The call returns `(WasCached: Bool, MeResponse: MeResponseV1)`. `MeResponseV1` extends `PlayerPublicProfile` (UE `ScriptStruct` `sps` inheritance), so `MeResponse.PlayerId` is the Prometheus ID. Once `WasCached == true`, cache the result, stop polling, emit the identity event.
- **Pros** — Conceptually simplest. Single call, single source of truth.
- **Cons** — If the local PMPlayerModel cache hasn't filled by the time the script starts polling, `WasCached == false` indefinitely (no upper bound on wait). **Plus a build-blocker discovered in the Pass-4 spike:** UE4SS in this build cannot call `GetCachedMeResponseV1` (or any other `(Bool out, X out)` UFunction on `PMPlayerModel`) from Lua at all — every documented placeholder shape errors at the marshaling layer. To make R-A work in this build the call would have to go through a BP wrapper (BP-side native UFunction calls bypass the marshaling block); see `docs/learnings/ue4ss-outparam-marshaling-failure.md`. Post-Pass-5, R-A is *more* expensive than R-B because R-B no longer requires a BP wrapper at all (see R-B's substrate revision).
- **Cost to build** — Was: small. Revised post-Pass-4: same as R-B's BP-wrapper cost + a poll loop. Revised post-Pass-5: **higher** than R-B (because R-B dropped its BP wrapper) — R-A now requires either a BP wrapper R-B doesn't OR a different cache-read path entirely (direct UProperty read, untested). Revised post-Pass-6 v2: R-A's gap widens further — Pass-6 v2 evidence shows `PMPlayerModel.WasCached` stays `false` across the identity-bootstrap window even while `PMPlayerPublicProfile.PlayerId` is populated, so R-A's premise (poll until `WasCached==true`) would mistake "cache not warm yet" for "identity not ready yet" and never resolve. R-B's walk-via-`PMPlayerPublicProfile` doesn't have this dependency.
- **Cost to change later** — Low — internal to `identity.lua`.

#### R-B — Event-driven (RegisterHook on Prometheus-side originating UFunction)

> **Substrate revision history.** Pre-Pass-4: "subscribe to delegate from Lua (likely with a Lua callback)." Post-Pass-4: "subscribe via `prop:Add(modActor, fname)` with a ModActor BP wrapper, since `prop:Add` requires a UObject target." Post-Pass-5: "RegisterHook on the originating Prometheus-side UFunction that drives the multicast delegate's broadcast" — because `prop:Add` itself is a silent no-op on this UE4SS build for `MulticastInlineDelegateProperty`. **Post-Pass-6 v2 (current): the concrete RegisterHook target is `PMIdentitySubsystem:GetIdentityState`, paired with a `FindAllOf("PMPlayerPublicProfile")` walk for the actual `PlayerId` read.** The walk is independent of `PMPlayerModel`'s `WasCached` flag (Pass-6 v2 evidence: ambient `PlayerId` populated on fire #9 with `WasCached=false`), so the read works inside any of the 4 identity-flow UFunctions Pass-6 v2 caught firing. `GetIdentityState` was picked because it fires earliest in the bootstrap window and is on `PMIdentitySubsystem` (one-instance-per-process, never re-instanced mid-session), making the hook lifecycle simpler than hooking a UFunction on `PMPlayerModel`. See `docs/learnings/os-runtime-data-model.md` (post-Pass-6 update) for the empirical evidence and `docs/learnings/ue4ss-multicast-delegate-add-silent-noop.md` for why the delegate-subscription path is unavailable. The ModActor BP wrapper substrate is no longer required for R-B.

- **What it is** — `RegisterHook(Pre|Post)` on `/Script/Prometheus.PMIdentitySubsystem:GetIdentityState` from Lua at module-init time. On hook fire, walk `FindAllOf("PMPlayerPublicProfile")` and read the first non-empty `prof.PlayerPublicProfile.PlayerId` — that's the local Prometheus ID. (At login / main-menu, the cache contains exactly one entry: the local player's own profile, just received as part of the same identity-bootstrap exchange that triggered the hook fire. Mid-match disambiguation, when other players' profiles populate the cache, falls back to joining on `PlayerState.PlayerNamePrivate` per the existing `findFriendlyNameByAccountId` walk in `identity.lua` — but the identity ADR resolves once-per-session at login, so the simple "first non-empty" read is sufficient under R-B's actual use-site.) On first successful read, cache the value and `UnregisterHook` (deferred one tick via `ExecuteInGameThread` so the in-flight engine call completes cleanly). **No polling loop. No BP wrapper. No delegate binding. No ongoing per-frame overhead after first resolution.**
- **Pros** — No polling. No BP work, no cook step, no Lua-BP bridge. Pure Lua. **Strictly cheaper substrate than the Pass-4 ModActor-BP-wrapper R-B.** Substrate is proven on this UE4SS build at three layers: (a) Pass-5 F6 confirmed `RegisterHook` registration succeeds; (b) Pass-6 v1+v2 confirmed registration scales cleanly to 79 UFunctions across `PMPlayerModel` + `PMIdentitySubsystem` with zero failures; (c) Pass-6 v2 captured 4 UFunctions firing during natural identity flow, with `PMPlayerPublicProfile` cache populated independently of `PMPlayerModel.WasCached`. The maintainer-recommended pattern for cross-actor BP↔Lua signaling on UE4SS today (per [Issue #455](https://github.com/UE4SS-RE/RE-UE4SS/issues/455)) — well-trodden path with years of community precedent. **Establishes `RegisterHook` on game-side UFunctions as the foundational reactive primitive** — every future event-driven feature that needs to react to game state (per-redirect events, end-of-match hooks, achievement triggers, remote-player profile arrivals via `GetDisplayNameV1` hooks) reuses the same primitive, no per-event BP authoring needed. Bypasses the entire class of vtable-mismatch and delegate-binding risks that broke the Pass-4 substrate. **Self-unhook after first resolution** keeps zero ongoing overhead — the hook is gone for the rest of the session.
- **Cons** — Hook target depends on engine-side UFunction call patterns rather than a stable contract — if a future game patch renames `GetIdentityState`, splits it, or stops calling it during the identity bootstrap, the hook target needs re-discovery (re-run Pass-6 against the patched build; the probe is preserved in `docs/features/pass2-probes/` exactly for this revisit case). The Pass-4 ModActor-BP-wrapper R-B had the same fragility at a different layer — neither approach is patch-proof. The walk-via-`PMPlayerPublicProfile` read is also context-dependent: it's the local player only at login / main-menu when no other profiles have been cached yet; in mid-session contexts (party joins, matchmaking, in-match) the cache contains remote profiles too and the "first non-empty" pick would be unreliable. R-B's actual use-site (resolve once at login) avoids this; the constraint is documented at the call site so future agents don't reuse the walk for mid-session re-resolution without adding disambiguation.
- **Cost to build** — Low. Pass-6 discovery probe complete (substrate end-to-end-validated); `mod/OSPlus/scripts/identity.lua` extension is ~50 lines: hook registration + walk + cache + deferred unhook + listener slot. Substrate amortizes across every future event-driven feature for free (no per-event setup beyond one `RegisterHook` call).
- **Cost to change later** — Low — internal to `identity.lua`. If UE4SS later fixes the `prop:Add` vtable issue and we want the delegate-payload struct directly, swap the hook for a delegate subscription with no schema or wire-format change required.

#### R-C — Hybrid (sync polling, delegate fallback after N retries)

- **What it is** — R-A by default. If `WasCached == false` after ~10 seconds (configurable), additionally subscribe to the delegate. Whichever path delivers the response first wins; the other deduplicates.
- **Pros** — Conceptually: fast happy path (warm cache) + robust slow path (cold start). Handles both scenarios without indefinite polling and without requiring delegate-binding for the common case.
- **Cons** — Most code. Two paths to test. Dedup logic required (don't emit identity event twice). **Two Pass-spike build-blockers, both unrecoverable on this UE4SS build:** (a) the warm-cache UFunction call doesn't work (Pass-4 — `(Bool out, X out)` marshaling block, see `docs/learnings/ue4ss-outparam-marshaling-failure.md`), and (b) the delegate-binding fallback doesn't work either (Pass-5 — `prop:Add` silent no-op, see `docs/learnings/ue4ss-multicast-delegate-add-silent-noop.md`). Post-Pass-5, R-C as originally framed has *no* viable arm on this build — both halves are blocked at the substrate layer. Reframing R-C onto Pass-5's RegisterHook substrate would just make it "R-B with a useless poll-loop bolted on."
- **Cost to build** — Was: medium-high. Revised post-Pass-5: **higher than R-B** for the same reason (any reframed R-C inherits R-B's substrate plus an unused poll loop) — and lower-bound is still gated on the same Pass-6 RegisterHook discovery probe.
- **Cost to change later** — Low.

### Trust posture — Sub-options (orthogonal to P/R)

- **T-α — Trust-on-claim.** Sidecar reports what the local game reports. No relay-side verification. Same posture as the archived position. **Pros:** zero infra cost. **Cons:** a modded client can spoof another player's Prometheus ID. The cost-to-attack is medium (requires Lua mod + relay protocol knowledge); the value-of-attack is low at MVP scope (no currency, no moderation, no exclusive grants).
- **T-β — Verified via Odyssey backend handshake.** Sidecar exchanges a Steam ticket / Odyssey OAuth token to validate the claimed Prometheus ID. **Pros:** spoof-resistant. **Cons:** requires reverse-engineering Odyssey's auth handshake (NDA territory per the Strikr-GG case in `os-prometheus-api-ecosystem.md`); creates a hard dependency on Odyssey's backend cooperation, **violating product anti-goal #3 ("survives the next Odyssey patch with zero work").** Not feasible at MVP.

## Decision

**Bind OSPlus profile rows to the Prometheus ID (Option P-B), resolved via the event-driven path (Option R-B), under trust-on-claim posture (Sub-option T-α).** SteamID retained as a secondary indexed column for cross-reference and as a fallback-display affordance.

Rationale, in order:
1. **P-B over P-A** — the platform-agnostic requirement is non-negotiable (maintainer-stated). P-A's "do this work later" cost is the same work P-B does now, plus a migration. Tracker-interop bonus is decisive among the close calls.
2. **P-B over P-C** — premature flexibility tax. P-C optimizes for a multi-platform world that isn't observed yet; if it materializes, the cost of adding a secondary identifier to a P-B schema is small.
3. **R-B over R-A** — Pre-Pass-4: "R-A is easier now, R-B compounds long-term." Post-Pass-4: "both pay the BP-wrapper cost; R-B uses it reactively." **Post-Pass-5: R-B is strictly cheaper than R-A** — R-B's substrate dropped the BP wrapper entirely (pivoted to `RegisterHook` on engine-side UFunctions, which works on this UE4SS build); R-A still requires either a BP wrapper or an untested direct-property-read path to route around the `(Bool out, X out)` marshaling block. R-A's failure mode (indefinite silent retry on a warm-cache check that may never succeed at the calling-convention layer) is also worse than R-B's (hook the originating UFunction once, fire-and-forget). **OSPlus's wedge surfaces are dominated by reactive game state** (per-redirect events, end-of-match hooks, achievement-condition triggers, remote-player profile arrivals); landing `RegisterHook` on engine-side UFunctions as the substrate at identity time means each of those features reaches for the established hook primitive instead of re-inventing either a poll loop or its own bridge.
4. **R-B over R-C** — R-C was "two paths so neither has to be solid." Post-Pass-4 it collapsed into "delayed R-B + extra dedup glue." **Post-Pass-5 it has no viable arm on this build** — the warm-cache half is blocked by the Pass-4 marshaling failure, and the delegate-fallback half is blocked by the Pass-5 silent-no-op finding. Any reframed R-C onto Pass-5's RegisterHook substrate is just R-B with a dead poll loop bolted on. Pay the RegisterHook substrate once; reuse forever.
5. **T-α over T-β** — T-β's dependency on Odyssey's backend violates anti-goal #3. The threat model at MVP doesn't justify that cost. Reopen for T-β if a future ADR introduces currency / trade / moderation.

**Conditional:** This ADR is acceptable for MVP and remains acceptable until any of the *Revisit triggers* below fires.

**Acceptance prerequisite — STATUS: MET (Pass-4 complete 2026-04-24; R-B substrate revised post-Pass-5 2026-04-25; revised substrate operationally validated via Pass-6 v2 2026-04-25 with concrete UFunction target identified).**

The acceptance question was: "does R-B have a viable substrate, or does the choice collapse to R-A vs R-C?" Three passes ran:

*Pass-4 (2026-04-24) — ModActor BP wrapper substrate.* Probe source `docs/features/pass2-probes/pass2_probes.lua` (F8 keybind, four iterative revisions through native-crash forensics). Established the `prop:Add(uobject, fname)` API surface and the BP-wrapper substrate plan. Documented in `docs/learnings/ue4ss-lua-multicast-delegate-binding.md` and the prior version of this ADR. **Verdict at the time: R-B substrate viable with BP-wrapper cost.**

*Pass-5 (2026-04-25) — end-to-end validation of the BP-wrapper substrate.* Probe source `docs/features/pass2-probes/pass2_probes.lua` (E1–E8 + E8.D suite, F1/F2/F3/F4/F6/F10 keybinds; ~10 iterative probe revisions). Goal: validate that the documented `prop:Add(modActor, "OnMeResponseFired")` substrate actually fires the bound UFunction during natural identity flow before committing the ADR to it. **Result: substrate is non-functional on this UE4SS build.** Six independent triangulation sub-tests in F10/E8.D ruled out shape-specific causes (cross-actor vs same-actor, FName vs string, alt API names) — `prop:Add` returns ok across every variation but `GetBindings()` reports 0 across every variation, and `Broadcast()` succeeds at marshaling but invokes nothing. Web research grounded the failure in source code analysis: most-likely root cause is a vtable-offset mismatch in UE4SS's parser for Omega Strikers' shipped `FMulticastInlineDelegateProperty` binary layout (PR #1073 added the Lua delegate API only ~5 months ago in production with no regression coverage for inline-multicast on non-engine-namespace properties on `UDataModel` subclasses). Full evidence chain in `docs/learnings/ue4ss-multicast-delegate-add-silent-noop.md`.

*Pass-5 pivot — RegisterHook on engine-side originating UFunction.* The maintainer-recommended workaround for cross-actor BP↔Lua signaling on UE4SS today is to hook the engine UFunction that *calls* `Broadcast` on the multicast delegate, not to subscribe to the delegate ([UE4SS Issue #455](https://github.com/UE4SS-RE/RE-UE4SS/issues/455)). For OSPlus's identity path this means `RegisterHook(Pre|Post)` on `PMPlayerModel:GetMeV1` (or whichever Prometheus-side UFunction Pass-6 identifies as the most reliable identity-flow fire), reading identity state from `self` inside the hook callback. **Substrate is proven at the registration layer on this UE4SS build** — Pass-5 F6 confirmed `Registered script hook (NN, NN) for Function …`. The remaining open question is *which* UFunction(s) to hook, which is what Pass-6 answers — a ~30min Lua probe with no BP/editor/cook work.

*Pass-6 v2 (2026-04-25) — operational validation + concrete UFunction target.* Probe source `docs/features/pass2-probes/pass2_probes.lua` (`NUM_SIX` keybind, E9 / `probeE9`; v2 install runs at module load via `NotifyOnNewObject` + `FindFirstOf` one-shot). Goal: identify which UFunction(s) on `PMPlayerModel` and `PMIdentitySubsystem` actually fire during natural identity flow, with what latency, and whether `self`-side identity data is reachable inside the hook callback. **Result: substrate is operationally proven; concrete target picked.** 79/79 hooks installed cleanly across `PMPlayerModel` (44) + `PMIdentitySubsystem` (35); zero `RegisterHook` failures. During cold-start login, exactly **4 UFunctions fired**: `PMIdentitySubsystem:GetIdentityState` (fires earliest, ~T+0 in the bootstrap window), `PMPlayerModel:GetCachedPlayerPublicProfile`, `PMPlayerModel:GetCachedPlayerMatchmakingConstraintsV1`, `PMIdentitySubsystem:HasFeatureFlag`. Critically: `FindAllOf("PMPlayerPublicProfile")` walk inside the callback returns the local `PlayerId` populated **independently of `PMPlayerModel.WasCached`** — fire #9 captured ambient `PlayerId` populated while `WasCached=false`, demonstrating that the cache-population path is decoupled from the `PMPlayerModel`-internal cache flag the original R-A would have polled. The 4 identity-flow UFunctions form an ~1.78s window during which the walk goes from "no profile in cache" to "local `PlayerId` populated"; any of the 4 would work as the trigger, but `PMIdentitySubsystem:GetIdentityState` is picked because (a) it fires earliest, (b) `PMIdentitySubsystem` is one-instance-per-process (no re-instancing concerns), (c) it's a natural semantic match for "identity changed, re-read it." `MeRequestV1Completed` itself was not caught by any of the 79 hooks, confirming Pass-5's finding that the multicast broadcast doesn't go through a regular UFunction call path on this build — but downstream pollers like `GetIdentityState` provide an equivalent reactive surface. The ModActor probe substrate is no longer required at any layer.

*Discovery output (final):*

- **Subscription substrate (D1) — Pass-4 BP-wrapper plan SUPERSEDED; Pass-5 RegisterHook pivot OPERATIONALLY VALIDATED.** Per Pass-5, `MulticastDelegateProperty:Add(uobject, fname)` is callable but does not register an engine-side binding on this UE4SS build. Per Pass-6 v2, `RegisterHook` on `PMIdentitySubsystem:GetIdentityState` fires reliably during natural identity flow, and the `FindAllOf("PMPlayerPublicProfile")` walk inside the callback yields the local `PlayerId` independently of `PMPlayerModel.WasCached`. **The chosen R-B substrate is concrete:** hook `GetIdentityState`, walk `PMPlayerPublicProfile` for `PlayerId`, cache + self-`UnregisterHook` on first non-empty read. Strictly cheaper than the Pass-4 BP-wrapper plan (no BP class, no cook step, no Lua-BP bridge) and validated end-to-end against the running game.
- **Cache pre-check substrate (D2) — not available via UFunction in this UE4SS build (Pass-4 finding, unchanged); empirically irrelevant per Pass-6 v2.** Spike swept three `PMPlayerModel` `(Bool out, X out)` UFunctions across all documented placeholder shapes: `GetCachedLinkCodeV1` (Bool + Str), `GetCachedPlayerPublicProfile` (Bool + Struct), `GetCachedMeResponseV1` (Bool + Struct). Every shape failed at the marshaling layer. Captured in `docs/learnings/ue4ss-outparam-marshaling-failure.md`. **Pass-6 v2 added a follow-on finding:** the cache-pre-check would have been *misleading* even if it had worked — `PMPlayerModel.WasCached` stays `false` across the identity-bootstrap window in which `PMPlayerPublicProfile.PlayerId` becomes readable. R-A / R-C polling on `WasCached` would have spun forever waiting for a flag that flips later than the data we actually need. R-B's walk-via-`PMPlayerPublicProfile` is independent of this flag.
- **Delegate signature mystery (D3, Pass-5 byproduct) — RESOLVED.** The signature for `MeRequestV1Completed` is package-scoped at `/Script/Prometheus.MeRequestV1Completed__DelegateSignature` (not class-scoped — explains why Pass-5 F3's `ForEachFunction` over `PMPlayerModel`'s class hierarchy found 0 delegate-signature candidates). Four parameters: `Succeeded: BoolProperty, RequestId: StrProperty, MeResponse: StructProperty, ErrorResponse: StructProperty`. Flags `0x130000` (`FUNC_Delegate | FUNC_Public | FUNC_MulticastDelegate`). This matters less for the pivoted R-B (which doesn't subscribe to the delegate) but is captured for any future work that targets this signature directly (e.g., a UE4SS C++ mod, or a future build where `prop:Add` works correctly). **Pass-6 v2 confirmation:** none of the 79 hooks caught the multicast broadcast itself, supporting the Pass-5 hypothesis that the broadcast doesn't go through a regular UFunction call path on this build. Downstream pollers like `GetIdentityState` provide the equivalent reactive surface, which is what R-B uses.

The original two-prereq frame ("D1 substrate exists?", "D2 substrate exists?") is now answered by all three passes combined: D1's *original* Pass-4 plan failed Pass-5's end-to-end test (silently) but R-B's *substrate* (event-driven identity from the running game) survives the Pass-5 pivot to RegisterHook AND the Pass-6 v2 operational validation, with strictly cheaper cost than the Pass-4 plan it replaces. D2 unavailable as designed AND empirically misleading (Pass-6 v2 `WasCached` finding); replaced by the walk-via-`PMPlayerPublicProfile` pattern in R-B's revised description, which doesn't depend on `WasCached`. R-A and R-C inherit both the UFunction-marshaling block AND the `WasCached`-misleading-flag finding — making R-B even more clearly the right answer post-Pass-6.

**Stage-5 prerequisite (blocks build, not sign-off) — STATUS: MET post-Pass-6 v2.**

*Original prereq* (Pass-4 era): "Which placeholder shape works for `GetCachedMeResponseV1`?" → Answered negative; warm-cache pre-check removed from R-B's scope.

*Pass-5 prereq update*: "Which `PMPlayerModel`-or-adjacent UFunction(s) fire reliably during natural identity flow such that `RegisterHook(Pre|Post)` on them gives us a Prometheus ID?" Pass-6 v2 answers this concretely: **`PMIdentitySubsystem:GetIdentityState`**, paired with a `FindAllOf("PMPlayerPublicProfile")` walk for the actual `PlayerId` read inside the callback. Three runner-up UFunctions also fire in the same identity-bootstrap window (`PMPlayerModel:GetCachedPlayerPublicProfile`, `PMPlayerModel:GetCachedPlayerMatchmakingConstraintsV1`, `PMIdentitySubsystem:HasFeatureFlag`); the chosen pick is `GetIdentityState` for three reasons: fires earliest, on a one-instance-per-process subsystem (`PMIdentitySubsystem`), and semantically matches "identity state changed, re-read it." All three runner-ups are documented in case a future patch removes `GetIdentityState`.

*Pass-6 v2 install timing footnote*: Pass-6 v1 had a probe-design bug where hooks were installed on `NUM_SIX` keypress (after login completed), missing every cold-start identity-flow event. Pass-6 v2 fixed this by installing at module-load via `NotifyOnNewObject("/Script/Prometheus.PMIdentitySubsystem", ...)` + a `FindFirstOf` one-shot for instances that already exist. The OSPlus production hook in `identity.lua` follows the same pattern: register at module load, not at user action. Captured in `docs/learnings/ue4ss-cold-start-hook-install-pattern.md`.

*Failure-mode contingency (no longer needed but kept for revisit)*: If a future Omega Strikers patch reshapes `PMIdentitySubsystem`'s identity-flow internals such that `GetIdentityState` no longer fires during the bootstrap window, the recovery path is "re-run Pass-6 against the patched build, pick a new target from the per-fire log, update the one `RegisterHook` call site in `identity.lua`." The four runner-up UFunctions Pass-6 v2 caught are leading candidates for the re-pick. Worst case (none of the four UFunctions fire either): re-open this ADR to choose between (a) the UE4SS C++ mod path (write a native mod that calls `FMulticastDelegateProperty::AddDelegate` via the engine API directly, bypassing the Lua vtable indirection) or (b) the `M`-style manual-relink path (Pass-2 confirmed `PlayerNamePrivate` from `PlayerState` works during character select; `M` would key on a manually-linked Prometheus ID resolved at first launch). Neither is needed today.

**Stage-5 build outcome — STATUS: PRODUCTION-VALIDATED post-v36 (2026-04-25).**

The Stage-5 build of `identity.lua` shipped in three iterations beyond the Pass-6-derived plan, all driven by a UE4SS-Lua-marshaling discovery that the Pass-6 evidence didn't surface (because Pass-6 only validated *that* hooks fire, not *what to do* inside the callback to read the actual ID):

| Version | Discovery | Outcome |
|---|---|---|
| v27→v32 | `FindAllOf("PMPlayerPublicProfile")`-walk path *did* work for getting the local Prometheus ID, but the disambiguation key was unreliable: the "first cached profile" heuristic resolved correctly on cold-start but mis-attributed under any non-cold-start condition (cross-validation against `PlayerState.PlayerNamePrivate` returned `"TGD I Greedom"` for a row that was actually `"Ispicas"` — a different player who'd cached profile fired earlier). The walk-and-disambiguate approach was abandoned in favor of direct subsystem introspection. | Pivoted to direct UFunction call on `PMIdentitySubsystem`. |
| v32 (probe) | UFunction enumeration on `PMIdentitySubsystem` revealed `GetAuthenticatedPlayerId(Valid: BoolProperty, OutPlayerId: StrProperty)` — exactly the canonical "is the local player authenticated, and what's their ID" call. Cleaner semantics than the walk path. | Switched target. |
| v33→v34→v35 | **Three failed call-shape attempts in a row** (`()`, `(false, "")`, `(false, nil)`) before landing on the correct UE4SS 3.0.1 multi-out-param shape: `inst:GetAuthenticatedPlayerId({}, {})`. Each iteration cost a full game-restart loop with the maintainer. The wrong shapes errored with diagnostic-but-misleading messages that *looked* like parameter-count problems but were actually about the marshaler's table-as-by-ref convention. The previous learning (`ue4ss-outparam-marshaling-failure.md`) had concluded "no Lua-side fix exists" for the `(Bool out, X out)` shape class, which biased the early v33 attempts toward "find a workaround" rather than "the previous learning's matrix never tested `({}, {})`." | v35 succeeded; production target locked in v36. |
| v36 (current) | Production-clean `readAuthenticatedPlayerId` using the validated `({}, {})` shape, with the `pluckOutParam` helper that defensively iterates buckets so a future UE4SS that splits multi-out-params correctly across buckets keeps working. Cross-validation block stripped; diagnostic logging removed. | Maintainer verified the resolved ID (`632680c154686dedd6522b09`) against their personal Odyssey account. **Production-shipping.** |

**The v33→v35 cost (three iterations, ~30min each) is a clean compounding cost for the codebase**: the new learning `docs/learnings/ue4ss-ufunction-out-param-marshaling-3-0-1.md` captures the canonical convention as a copy-pasteable reference; `lua-conventions.mdc`'s "Lua-not-C++ reflex" section now has a worked-and-confirmed reference instead of an unresolved one; and the older `ue4ss-outparam-marshaling-failure.md` is annotated as partially superseded with the corrected coverage matrix. Future agents reaching for any `(Out X, Out Y)` UFunction call will hit the canonical shape directly.

**Substrate-validation meta-lesson, in addition to the three-pass spike pattern already documented in Notes:** the Pass-6 acceptance probe validated that the hook fires reliably, but did *not* validate the data-extraction call shape inside the callback. The implicit assumption was "if we can find the right hook, the rest is mechanical UFunction calls" — which collapsed when the canonical UFunction (`GetAuthenticatedPlayerId`) had a calling-convention problem the previous Pass-4 spike had concluded was unsolvable for its shape class. **Going forward: any acceptance probe for an event-driven substrate should include a smoke test of the data-extraction call inside the callback, not just confirmation that the callback fires.** Pass-6 should have done a `pcall(function() inst:GetAuthenticatedPlayerId({}, {}) end)` smoke test inside one of the 79 hooks it installed; that would have caught the marshaling-shape question while still in probe mode and saved the v33→v35 game-restart loops.

## Consequences

**What this commits us to:**

- ~~A Pass-6 RegisterHook discovery probe~~ — **complete (Pass-6 v1 + v2, 2026-04-25).** Probe lives in `docs/features/pass2-probes/pass2_probes.lua` (`NUM_SIX` keybind, `probeE9`); install pattern is `NotifyOnNewObject` + `FindFirstOf` one-shot at module load. Result documented in *Acceptance prerequisite* above and in `docs/learnings/os-runtime-data-model.md` (Pass-6 v2 update). Probe is preserved (not deleted) so future game patches that reshape the identity-flow UFunction graph can be re-mapped via the same `NUM_SIX` keypress + log-and-pick workflow. **No BP work, no cook step, no Lua-BP bridge** — Pass-4-era ModActor BP wrapper commitment replaced entirely.
- `mod/OSPlus/scripts/identity.lua` extends to: (1) `RegisterHook(Pre|Post)` on `/Script/Prometheus.PMIdentitySubsystem:GetIdentityState` at module init, (2) on hook fire, walk `FindAllOf("PMPlayerPublicProfile")` and read the first non-empty `prof.PlayerPublicProfile.PlayerId` (independent of `PMPlayerModel.WasCached`), (3) on first successful read, cache the value on the module + fan out to subscribers via `M.onPrometheusIdResolved(fn)` registration API + defer one tick via `ExecuteInGameThread` and `UnregisterHook` to release the hook (no ongoing per-frame overhead after first resolution), (4) expose `M.getLocalPrometheusId()` for synchronous post-resolution reads. The new resolver coexists with the existing `M.resolveSteamId()` + `M.resolveDisplayName()` paths; it doesn't replace them, it adds the canonical Prometheus ID alongside. **No polling loop, no warm-cache pre-check, no BP wrapper, no `WasCached` dependency.**
- The `RegisterHook`-on-engine-UFunction pattern becomes the **foundational reactive primitive for the codebase.** Future features that need to react to game state (per-event redirect capture on `PMRockCharacter:LastRedirectKnockBack`, end-of-match hooks for `in-game-profile-mvp`'s capture surface, achievement-condition triggers, remote-player profile arrivals via `GetDisplayNameV1` hooks) reuse the same primitive — adding a new event source is "one new `RegisterHook` call", no per-event BP authoring, no per-event substrate build. The cold-start install pattern (`NotifyOnNewObject` + `FindFirstOf` one-shot) is also reusable for any feature that needs to hook a UObject not guaranteed to exist at mod-load time. This is also the maintainer-recommended pattern per [UE4SS Issue #455](https://github.com/UE4SS-RE/RE-UE4SS/issues/455). Captured in `docs/learnings/ue4ss-multicast-delegate-add-silent-noop.md` (the "why" of the pivot) + `docs/learnings/ue4ss-cold-start-hook-install-pattern.md` (the "how" of cold-start install timing).
- Sidecar wire format (and relay handshake) carries `prometheusId` as the canonical player key. Existing `steamId`-keyed wire formats are deprecated; one migration step happens during Stage-5 build of `in-game-profile-mvp`. The migration plan lands in `0002-profile-storage`.
- Profile row schema starts with: `prometheusId TEXT PRIMARY KEY`, `steamId TEXT NULL` (indexed), `displayName TEXT`, `currentPlatform TEXT` (Steam / etc.), `logoId / nameplateId / emoticonId / titleId TEXT` (cosmetic loadout — surfaced from the same `PMPlayerModel` state the identity hook reads, useful for the eventual unlockable feature without an extra round-trip), `createdAt`, `updatedAt`. Exact column types and storage location land in `0002-profile-storage`.
- Cross-tracker interop becomes possible (any future feature that wants to enrich a profile with `clarioncorp.net` / `strikr.gg` data has the joinable key without a new resolution layer).
- A profile row can exist for a player who has never connected via Steam — `steamId` may be `NULL`.

**What this rules out — until a future ADR supersedes this:**

- Anti-impersonation guarantees. Any feature whose security model depends on Prometheus ID being authentic is out of scope until a follow-up ADR introduces T-β.
- Treating SteamID as the primary key in any subsystem. Cross-reference only.
- "Live re-resolve mid-match" patterns. Identity resolves once per OSPlus session at startup. Mid-session changes (player logs out and back in within the same OSPlus run) are not handled — would require an explicit re-resolve hook.
- Composite-key flexibility. Adding a second primary identifier (e.g., Epic ID) later requires a small schema change but doesn't require ADR supersession (the ADR commits to *Prometheus ID as primary*; secondary indexed columns are policy under it).

**Revisit triggers:**

- First feature whose threat model requires authenticated identity (currency / trading / moderation actions / earned exclusivity gated by identity authenticity).
- Odyssey ships an official auth or identity API (cheap path to T-β opens up).
- A confirmed in-the-wild spoofing case affecting a real OSPlus user.
- The first non-Steam OS launch is observed in OSPlus telemetry — not a trigger for revisiting *this* decision (it confirms the platform-agnostic requirement was load-bearing); worth recording as ADR-validation evidence.
- An Omega Strikers patch reshapes `PMIdentitySubsystem`'s or `PMPlayerModel`'s identity-flow internals such that `PMIdentitySubsystem:GetIdentityState` no longer fires during the bootstrap window. Cheap recovery: re-run Pass-6 against the patched build, pick a new target from the four runner-up UFunctions (`GetCachedPlayerPublicProfile`, `GetCachedPlayerMatchmakingConstraintsV1`, `HasFeatureFlag`) or whatever new ones surface, update the one `RegisterHook` call site in `identity.lua`. Doesn't reopen this ADR — only the implementation file. (The `PlayerPublicProfile` walk path is much more durable than any single hook target — Prometheus's wire-level types likely outlive any specific local UFunction.)
- Worst case (none of the four UFunctions fires either, AND `FindAllOf("PMPlayerPublicProfile")` no longer returns the local profile at login): ADR re-opens to choose between the UE4SS C++ mod path (write a native mod that calls `FMulticastDelegateProperty::AddDelegate` via the engine API directly, bypassing UE4SS Lua's vtable indirection) and the M-style manual-relink path (key on a manually-linked Prometheus ID resolved at first launch via `PlayerNamePrivate` from `PlayerState` during character select, which Pass-2 confirmed works). Pre-Pass-6 evidence (79 enumerated UFunctions plus the engine's necessity to invoke *something* during login to populate the cache) made this scenario unlikely; Pass-6 v2's actual results (4 firing UFunctions + populated `PMPlayerPublicProfile.PlayerId` independent of `WasCached`) make it even less likely.
- A UE4SS update fixes the `MulticastInlineDelegateProperty:Add` silent-no-op characterized in Pass-5 (likely a vtable-layout fix for the UE 4.27 template) — opens up a cheaper path for *future* features that want the delegate-payload struct directly (does not invalidate the present R-B choice, since `RegisterHook` on engine UFunctions is still right on its merits, but reopens the "subscribe-to-delegate" sub-decision if a follow-up feature wants the full struct without a `self`-side property read).
- A UE4SS update fixes the `(Bool out, X out)` UFunction-marshaling failure characterized in Pass-4 — opens a cheaper path for any *future* synchronous cache-read need (does not invalidate the present R-B choice for identity, since R-B's reactive substrate is still right on its merits, but reopens the "warm-cache pre-check" sub-decision if a follow-up feature wants it).

## Notes

- The Pass-3 finding that `MeResponseV1` extends `PlayerPublicProfile` (`sps` chain in the UObject dump) is what makes the resolution path cheap. Without that inheritance, getting both Prometheus ID and the cosmetic loadout would require multiple calls. Don't lose sight of this in implementation — call once, populate everything.
- **Maintainer-stated principle that informed R-B over R-A:** "We should always prefer event-driven over polling." This ADR records the principle's first concrete application but does not codify it as a project-wide rule — that would be `AGENTS.md` / a rules-file change, separate scope. If the principle holds across two or three more ADRs, it's worth elevating then. Note that Pass-5's pivot kept R-B as event-driven *and* made it cheaper than R-A (which is the durable evidence the principle was right; the substrate-cost penalty the original R-B paid as the cost of being event-driven turned out to evaporate post-pivot).
- The pre-spike R-B implementation included a subscribe-time cache pre-check (`GetCachedMeResponseV1` once, then subscribe) as a defensive warm-start. The Pass-4 spike falsified that path's availability in this UE4SS build (see *Acceptance prerequisite*); R-B's Pass-4-era revised posture was "subscribe → wait for natural login fire." The Pass-5 pivot replaced "subscribe" with "RegisterHook on the originating UFunction" — same reactive substance, different mechanism, strictly cheaper substrate. Re-introducing a warm-cache pre-check is still a future option (BP-routed UFunction call, direct property read, or UE4SS-upgrade path), tracked in *Revisit triggers*.
- The Pass-4 + Pass-5 + Pass-6 spike output is captured across four learnings:
  - `docs/learnings/ue4ss-multicast-delegate-add-silent-noop.md` — **Pass-5 finding (load-bearing):** the documented `prop:Add(uobject, fname)` API silently no-ops on this UE4SS build for `MulticastInlineDelegateProperty`; likely root cause is a vtable-offset mismatch in UE4SS's binary parser. Maintainer-recommended workaround: `RegisterHook` on the originating engine UFunction (UE4SS Issue #455). Carries the full Pass-5 evidence chain and source-level web-research grounding.
  - `docs/learnings/ue4ss-cold-start-hook-install-pattern.md` — **Pass-6 v1 → v2 finding (load-bearing for the implementation):** the `RegisterHook` registration must happen at module load (via `NotifyOnNewObject` + `FindFirstOf` one-shot), not on a user keypress, for any UFunction that fires during cold-start before any user input. The Pass-6 v1 keypress-install probe missed every login-time `MeRequestV1` fire; the v2 module-load install caught all four identity-flow UFunctions cleanly. The OSPlus production hook follows the v2 pattern exactly.
  - `docs/learnings/ue4ss-lua-multicast-delegate-binding.md` — **Pass-4 finding (API surface, no longer load-bearing for ADR 0001):** the corrected `prop:Add(uobject, fname)` API + the Lua-callback false-friend trap (Rev-3 native crash) + the metatable-hidden-by-`__index` introspection trap. The `__index` placeholder trap and the crash-survivable logging pattern remain transferable to *all* UE4SS userdata work, not just delegates.
  - `docs/learnings/ue4ss-outparam-marshaling-failure.md` — **Pass-4 finding (still load-bearing):** the `(Bool out, X out)` UFunction marshaling block in this UE4SS build, with the specific error patterns per placeholder shape so future agents don't re-RE the failure. Workarounds documented. Affects Pass-6's choice of which UFunctions are useful to hook (`self`-side property reads only, no Lua-issued out-param calls inside hook callbacks). Pass-6 v2 added a corollary: `PMPlayerModel.WasCached` is also misleading at the moment `PMPlayerPublicProfile.PlayerId` is populated, so the walk-via-`PMPlayerPublicProfile` path is doubly preferable to any `PMPlayerModel`-cache-flag approach.
- If Stage 5 discovers `MeResponseV1.PlayerId` is empty for any local player who has bypassed the Odyssey EULA flow (e.g., first-launch before consent), this ADR may need a sub-decision on synthesizing a stable per-install random ID for "guest" rows. Not anticipated; flagged as a discovery scenario rather than a pre-decided branch.
- The composite-key option (P-C) is honestly tempting for a "we'll regret hardcoding the schema" gut feeling — but the rule is "two real options with honest pros and cons" and P-C's *honest* con is that it solves a problem no observed user has yet. Naming that explicitly so a future ADR doesn't dust it off as "the obvious flexible answer we should have picked."
- **Three-pass spike pattern as a meta-lesson for ADR substrate validation.** Pass-4 stopped at "the documented API exists and accepts our call shape" — sufficient for the ADR's "API surface" question, *insufficient* for "the API actually works on this game's binary." Pass-5 caught a substrate that would have failed at Stage-5 build, rescued the ADR with a strictly-cheaper pivot, and produced a transferable learning about a UE4SS class of bugs (vtable-offset mismatches in virtual-method-dispatch APIs presenting as silent no-ops). Pass-6 v1 then **caught a probe-design bug in its own first iteration** — keypress-installed hooks couldn't catch cold-start events, and the v1 "0 fires" result would have been misread as "no UFunctions fire during identity flow" (false-negative substrate failure) had we not noticed the timing gap. Pass-6 v2 fixed the install timing and operationally validated R-B end-to-end. Going forward: (a) any ADR whose substrate depends on a UE4SS Lua API method against a non-engine-namespace property should include a 2-line `Add → GetBindings`-style observable-state validation in its acceptance prereq; (b) any probe that depends on event timing should validate that its install path can capture the *earliest* event the feature cares about (not just *any* event of that shape) — for cold-start events, this means module-load install via `NotifyOnNewObject`, not user-action install. The composite cost was ~12 probe iterations across 7 keybinds; the benefit was the BP-wrapper substrate work avoided AND the operational R-B substrate validated AND two reusable UE4SS lessons extracted (vtable-mismatch silent no-op + cold-start install timing).
